🎯 LeetCode 精讲 #76 - 最小覆盖子串

📋 题目描述

来源: LeetCode 第76题
难度: Hard
链接: https://leetcode.cn/problems/minimum-window-substring/

题目原文

给你一个字符串 s 、一个字符串 t 。返回 s 中涵盖 t 所有字符的最小子串。如果不存在涵盖 t 所有字符的子串，则返回空字符串 "" 。

注意：

  • 对于 t 中重复字符，我们寻找的子字符串中该字符数量必须不少于 t 中该字符数量。
  • 如果 s 中存在这样的子串，我们保证它是唯一的答案。

示例

示例 1:

输入: s = "ADOBECODEBANC", t = "ABC"
输出: "BANC"
解释: 最小覆盖子串 "BANC" 包含来自字符串 t 的 'A'、'B' 和 'C'。

示例 2:

输入: s = "a", t = "a"
输出: "a"
解释: 整个字符串 s 是最小覆盖子串。

示例 3:

输入: s = "a", t = "aa"
输出: ""
解释: t 中两个字符 'a' 均应包含在 s 的子串中，因此没有符合条件的子串，返回空字符串。

约束条件

  • 1 <= s.length, t.length <= 10^5
  • s 和 t 由英文字母组成

💡 解题思路

核心思想：滑动窗口

这道题是**滑动窗口（Sliding Window）**算法的经典应用。滑动窗口的本质是维护一个可变大小的窗口，通过左右指针的移动来高效地寻找满足条件的子串。

为什么用滑动窗口？

暴力解法是枚举 s 的所有子串，检查是否包含 t 的所有字符，时间复杂度 O(n³) 或 O(n²)，对于 10^5 的数据规模无法接受。

滑动窗口可以将时间复杂度降到 O(n)，因为每个字符最多被访问两次（左指针和右指针各一次）。

算法步骤详解

第一步：统计 t 中字符需求

用哈希表 need 记录 t 中每个字符需要的数量。

例如 t = "ABC"：

need = {'A': 1, 'B': 1, 'C': 1}

例如 t = "AABC"：

need = {'A': 2, 'B': 1, 'C': 1}

第二步：维护窗口状态

用哈希表 window 记录当前窗口中各字符的实际数量。

第三步：双指针移动

  • 右指针 right：扩展窗口，尝试包含更多需要的字符
  • 左指针 left：收缩窗口，尝试找到最小覆盖子串

第四步：判断是否满足条件

需要知道窗口中是否已经包含了 t 中所有字符（且数量足够）。

这里用一个变量 valid 记录「满足数量要求的字符种类数」。

当 valid == len(need) 时，说明当前窗口已覆盖 t。

第五步：收缩窗口找最优解

当窗口满足条件时，尝试移动左指针收缩窗口：

  1. 如果当前窗口比已记录的最小窗口还小，更新答案
  2. 移动左指针，将移出窗口的字符从 window 中减去
  3. 如果移出的字符刚好满足数量下限，valid 减1

详细流程示例

以 s = "ADOBECODEBANC", t = "ABC" 为例：

初始化: need = {'A':1, 'B':1, 'C':1}, window = {}, valid = 0

right=0: s[0]='A', window={'A':1}, valid=1 (A满足) → 窗口[A], 不满足全部
right=1: s[1]='D', window={'A':1,'D':1}, valid=1 → 窗口[AD], 不满足
right=2: s[2]='O', window={'A':1,'D':1,'O':1}, valid=1 → 窗口[ADO], 不满足
right=3: s[3]='B', window={'A':1,'B':1,'D':1,'O':1}, valid=2 (B满足) → 窗口[ADOB], 不满足
right=4: s[4]='E', window加E, valid=2 → 窗口[ADOBE], 不满足
right=5: s[5]='C', window={'A':1,'B':1,'C':1,...}, valid=3 (全部满足!) → 窗口[ADOBEC]

此时valid==3，开始收缩left:
left=0: 移出'A', window['A']=0, valid=2, 当前窗口"ADOBEC"长度6，记录答案
left=1: 移出'D', window['D']=0, 但D不在need中，valid仍为2
...

继续扩展right直到再次满足条件，再收缩，最终找到"BANC"

📝 参考代码

class Solution:
    def minWindow(self, s: str, t: str) -> str:
        from collections import defaultdict
        
        # 记录t中每个字符的需求数量
        need = defaultdict(int)
        for char in t:
            need[char] += 1
        
        # 记录当前窗口中各字符的实际数量
        window = defaultdict(int)
        
        # 左右指针
        left = 0
        right = 0
        
        # valid记录满足数量要求的字符种类数
        valid = 0
        
        # 记录最小覆盖子串的起始位置和长度
        start = 0
        length = float('inf')
        
        # 右指针扩展窗口
        while right < len(s):
            # c是即将移入窗口的字符
            c = s[right]
            right += 1  # 扩大窗口
            
            # 如果c是需要的字符，更新窗口数据
            if c in need:
                window[c] += 1
                # 当窗口中该字符数量刚好满足需求时，valid增加
                if window[c] == need[c]:
                    valid += 1
            
            # 当窗口满足条件（包含t所有字符）时，尝试收缩左边界
            while valid == len(need):
                # 更新最小覆盖子串
                if right - left < length:
                    start = left
                    length = right - left
                
                # d是即将移出窗口的字符
                d = s[left]
                left += 1  # 收缩窗口
                
                # 如果d是需要的字符，更新窗口数据
                if d in need:
                    # 当窗口中该字符数量刚好不满足需求时，valid减少
                    if window[d] == need[d]:
                        valid -= 1
                    window[d] -= 1
        
        # 返回结果
        return "" if length == float('inf') else s[start:start + length]

⏱️ 复杂度分析

维度 复杂度 说明
时间复杂度 O(|s| + |t|) 左右指针各遍历s一次，初始化need遍历t一次
空间复杂度 O(|\Σ|) 哈希表存储字符计数，Σ是字符集（本题为英文字母，大小固定）

时间复杂度详细分析：

  • 右指针 right 从0移动到 len(s)，每个字符最多被访问一次
  • 左指针 left 从0移动到 len(s)，每个字符最多被访问一次
  • 因此时间复杂度为 O(2|s|) = O(|s|)
  • 初始化 need 需要 O(|t|)
  • 总时间复杂度：O(|s| + |t|)

🔑 关键要点

  1. 滑动窗口的适用场景

    • 找满足某条件的连续子串/子数组
    • 条件是可累加/可维护的（如字符计数、和、最大值等）

  2. 双指针的配合逻辑

    • 右指针负责「扩张」：寻找可行解
    • 左指针负责「收缩」：优化可行解
    • 当窗口满足条件时，才移动左指针

  3. valid 变量的妙用

    • 不用每次都遍历 need 检查是否满足条件
    • 用计数器维护满足条件的字符种类数
    • 时间从 O(|Σ|) 优化到 O(1)

  4. 边界处理

    • 字符不存在于 need 中时，直接忽略
    • 窗口中字符数量超过需求时，不影响 valid
    • 只有当数量刚好等于需求时，才增减 valid

  5. 常见错误

    • 忘记处理 t 中有重复字符的情况
    • valid 的判断条件写错（应该是 == 而不是 >=）
    • 收缩窗口时先减 valid 还是先移出字符的顺序问题

🎓 拓展思考

Q1: 如果要求「覆盖子串」中字符的相对顺序与 t 相同，该怎么做？

提示：这不再是滑动窗口问题，需要考虑动态规划（最长公共子序列的变种）或双指针匹配。

Q2: 如何优化空间复杂度到 O(1)？

提示：由于字符集是固定的英文字母（52个大小写字母 或 128个ASCII），可以用固定大小的数组代替哈希表。

Q3: 如果 s 和 t 非常大（10^7），且只能顺序读取 s，如何优化？

提示：考虑使用「最小堆」或「有序映射」来维护窗口状态，或者采用多趟扫描策略。
